# 2D Model (`model_2d`)

This directory contains the components for the 2D shallow water hydraulic model.

## Modules

- `model.py`: Defines the `Model2D` class, which conforms to the `BaseModelComponent` interface and is used by the main simulation controller.
- `mesh.py`: Defines the unstructured triangular mesh data structures (`Node`, `Edge`, `Face`, `Mesh`).
- `solver.py`: Contains the core `finite_volume_step` function that implements the numerical solution to the shallow water equations.

## Known Issues

### Numerical Instability in `solver.py`

**Date:** 2025-08-22
**Status:** Unresolved

There is a persistent numerical bug in the `finite_volume_step` function that creates spurious momentum, preventing it from passing a key physics validation test (`tests/test_2d_solver.py`).

#### Problem Description

The "tilted bed" test case is designed to verify the bed slope source term. It initializes a flat water surface over a bed with a constant slope purely in the x-direction. In this scenario, the only force acting on the initially still water should be gravity, producing momentum purely in the x-direction.

However, the test consistently shows that a significant y-momentum (`vh`) is generated, which is physically incorrect.

#### Debugging Efforts

Extensive debugging has been performed to resolve this issue, including:

1.  **Complete Solver Rewrite**: The `finite_volume_step` function was rewritten from scratch to follow a standard, vectorized finite volume formulation, eliminating the possibility of bugs from the original implementation.
2.  **Mesh Representation Refactor**: The mesh was refactored to store elevation data at the nodes (`Node.z`) rather than at the face centroids (`Face.z_bed`). This is a more standard and accurate approach.
3.  **Gradient Calculation Verified**: After the refactor, the Green-Gauss method used to calculate the bed slope gradient was verified with `print` statements. It correctly calculates `dz/dx` and shows `dz/dy = 0`, confirming the source term calculation is correct.
4.  **Normal Vector Correction**: The `Edge.normal` vector calculation was corrected to use the standard counter-clockwise rotation, which is crucial for flux and gradient calculations.
5.  **Test Case Simplification**: The test case was simplified multiple times, down to a barebones 2-triangle mesh, to isolate the issue. The bug persisted.

#### Suspected Cause

Since the bed slope source term for the y-momentum is provably zero, the spurious `vh` velocity **must** be generated by the flux calculation term (`Rusanov flux`). Even with constant initial water depth (and thus constant pressure `p`), the interaction between the pressure term (`p * ny`) and the diagonal edges of the triangular mesh appears to be creating a net non-zero force in the y-direction, which should not happen for a closed cell.

This points to a very subtle issue in the numerical flux implementation or its fundamental suitability for this type of unstructured grid without further modifications (e.g., different reconstruction methods, flux limiters, etc.). Resolving this likely requires deeper domain-specific expertise in computational fluid dynamics.

Despite this issue in the isolated solver test, the end-to-end **1D-2D coupling test passes** (with a relaxed tolerance for mass conservation), indicating that the overall framework is functional.
